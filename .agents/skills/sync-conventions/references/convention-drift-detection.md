# 컨벤션 드리프트 감지 가이드

`/sync-conventions`가 기존 `CONVENTIONS.md`를 업데이트할 때 사용하는 판단 규칙이다.

---

## 기본 원칙

1. 기존 `CONVENTIONS.md`가 있으면 프로젝트 전체를 다시 스캔하지 않는다
2. 먼저 변경된 파일과 git diff를 본다
3. 반복적이고 의도적인 변화만 팀 규칙 후보로 본다
4. Custom Rules는 항상 최우선이며 자동으로 덮어쓰지 않는다
5. `sync-architecture`나 `ARCHITECTURE.md`에 의존하지 않는다

---

## 우선 보는 신호

1. staged / unstaged diff
2. 새로 생성되거나 삭제된 파일
3. 최근 대화에서 AI가 수정한 파일
4. 사용자가 직접 리팩토링한 코드
5. 같은 패턴이 반복된 인접 파일

---

## 사용자 의도 후보 판단

다음 경우 사용자 의도 후보로 간주한다:

- AI가 만든 구조를 사용자가 다른 구조로 일관되게 재정리함
- helper 추출, naming, hook/service 분리 방식이 반복적으로 바뀜
- 기존 프로젝트 스타일에 더 가까운 방향으로 AI 코드가 수정됨
- 단순 포맷 수정이 아니라 설계 선택이 바뀜

다음 경우는 사용자 의도 후보로 보지 않는다:

- import 순서만 정리됨
- formatter가 자동 정리한 수준의 변경
- 단발성 오타 수정

---

## 질문이 필요한 패턴

- AI 패턴과 사용자 패턴이 충돌하지만 둘 다 타당함
- 기존 컨벤션 파일과 달라졌고, 새 패턴이 반복될 가능성이 있음
- cleanup / refactor 과정에서 새로운 정리 기준이 드러남

질문 예시:

```markdown
💡 사용자 의도 후보를 감지했습니다.

- 이전 패턴: [A]
- 현재 패턴: [B]

이 변경을 코딩 컨벤션에 반영할까요?
→ yes / no / modify [내용]
```

---

## 자동 반영 가능한 패턴

질문 없이 반영 가능한 경우:

- 기존 컨벤션 파일의 빈 항목을 실제 코드 패턴으로 채우는 경우
- 여러 변경 파일에서 동일한 패턴이 명확히 반복되는 경우
- 기존 규칙보다 더 구체적인 실제 패턴이 드러난 경우

---

## 업데이트 단위

전체 파일 재작성보다 **섹션 단위 갱신**을 우선한다.

예시:

- 테스트 컨벤션 > Mock 패턴
- 소스 코드 컨벤션 > 클래스/함수 패턴
- 소스 코드 컨벤션 > 에러 처리
- 리팩토링 기준
- 안티패턴
