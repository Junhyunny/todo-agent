# 인수 기준(AC) 작성 가이드

인수 기준(Acceptance Criteria)은 스토리가 "완료"된 상태를 사용자 관점에서 정의한다.
개발자가 자동화 테스트를 작성하고, PM이 인수 테스트에 사용하는 기준이 된다.

---

## 기본 형식

```
GIVEN [사전 조건 — 테스트가 시작될 때의 상태]
WHEN  [사용자 행동 — 무엇을 했을 때]
THEN  [기대 결과 — 어떤 일이 일어나야 하는가]
```

### 각 구성요소 설명

| 구성요소 | 역할 | 작성 원칙 |
|---------|------|---------|
| **GIVEN** | 테스트 시작 시점의 상태 | 구체적인 사전 조건 (로그인 여부, 데이터 존재 여부 등) |
| **WHEN** | 사용자가 수행하는 행동 | 단일 행동으로 표현 |
| **THEN** | 행동 후 기대되는 결과 | 검증 가능하고 측정 가능하게 작성 |

---

## 작성 원칙

### 1. 테스트 가능해야 한다

AC는 자동화 테스트로 작성할 수 있어야 한다.
"사용자가 만족한다"처럼 측정 불가능한 표현은 피한다.

### 2. 한 AC에 하나의 시나리오

여러 행동과 결과를 하나의 AC에 섞지 않는다.

### 3. 해피 패스 먼저, 예외 케이스 나중

정상 흐름을 먼저 정의한 후 엣지 케이스와 오류 케이스를 추가한다.

### 4. 사용자 관점으로 작성

기술 구현 방식이 아닌, 사용자가 경험하는 것을 기준으로 작성한다.

---

## 예시 비교

### 해피 패스 (정상 흐름)

```
✅ 좋은 예

GIVEN 제안서 요약 페이지를 방문했을 때
WHEN  "PDF 다운로드" 버튼을 클릭하면
THEN  영업 제안서의 PDF 파일이 내 컴퓨터에 다운로드된다
```

```
❌ 나쁜 예

GIVEN 사용자가 로그인했을 때
WHEN  버튼을 클릭하고 API가 호출되면
THEN  파일이 생성되어 저장된다
```

나쁜 예의 문제:
- GIVEN이 너무 광범위함 (어떤 페이지인지 불명확)
- WHEN에 기술 구현(API 호출)이 포함됨
- THEN이 사용자 관점이 아닌 시스템 관점

## AC 작성 순서

### 1단계: 해피 패스

사용자가 올바른 입력으로 원하는 작업을 완료하는 시나리오.

```
GIVEN 유효한 이메일과 비밀번호로 로그인 페이지를 방문했을 때
WHEN  로그인 버튼을 클릭하면
THEN  대시보드 페이지로 이동하고 내 이름이 표시된다
```

### 2단계: 엣지 케이스

경계값, 빈 상태, 대량 데이터 등 특수한 상황.

```
GIVEN 주문 목록이 비어 있는 상태에서
WHEN  주문 목록 페이지를 방문하면
THEN  "아직 주문 내역이 없습니다" 메시지가 표시된다
```

### 3단계: 오류 케이스

잘못된 입력, 권한 부족, 시스템 오류 등.

```
GIVEN 이메일 형식이 올바르지 않은 값을 입력했을 때
WHEN  로그인 버튼을 클릭하면
THEN  "올바른 이메일 형식을 입력해주세요" 오류 메시지가 표시된다
```

## 완성된 AC 세트 예시

### 스토리: 영업 제안서 PDF 다운로드

```
AC 1 (해피 패스)
GIVEN 제안서 요약 페이지를 방문했을 때
WHEN  "PDF 다운로드" 버튼을 클릭하면
THEN  영업 제안서의 PDF 파일이 내 컴퓨터에 다운로드된다

AC 2 (파일명)
GIVEN 제안서 요약 페이지를 방문했을 때
WHEN  "PDF 다운로드" 버튼을 클릭하면
THEN  다운로드된 파일의 이름은 "proposal-[고객사명]-[날짜].pdf" 형식이다

AC 3 (로딩 상태)
GIVEN PDF 생성에 시간이 걸리는 경우
WHEN  "PDF 다운로드" 버튼을 클릭하면
THEN  로딩 스피너가 표시되고 버튼이 비활성화된다

AC 4 (오류 처리)
GIVEN PDF 생성 중 서버 오류가 발생했을 때
WHEN  다운로드를 시도하면
THEN  "PDF 생성에 실패했습니다. 다시 시도해주세요" 메시지가 표시된다
```

---

## 자주 하는 실수

| 실수 | 예시 | 해결 방법 |
|------|------|---------|
| 검증 불가한 결과 | `THEN 사용자가 만족한다` | 구체적인 UI 변화로 표현 |
| 여러 행동 묶기 | `WHEN 버튼을 클릭하고 입력하면` | AC를 분리 |
| 기술 구현 포함 | `WHEN API가 200을 반환하면` | 사용자 행동으로 표현 |
| GIVEN 생략 | GIVEN 없이 WHEN부터 시작 | 항상 명확한 사전 조건 명시 |
| 너무 광범위한 THEN | `THEN 모든 것이 정상 작동한다` | 구체적으로 무엇이 어떻게 되는지 명시 |

## TrackerBoot 등록 형식

TrackerBoot의 `tasks` 배열에 각 AC를 한 줄로 매핑한다.

```json
{
  "tasks": [
    {
      "description": "GIVEN 제안서 요약 페이지를 방문했을 때 WHEN 'PDF 다운로드' 버튼을 클릭하면 THEN PDF 파일이 다운로드된다"
    },
    {
      "description": "GIVEN PDF 생성 중 오류 발생 시 WHEN 다운로드를 시도하면 THEN 오류 메시지가 표시된다"
    }
  ]
}
```
