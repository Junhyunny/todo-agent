# 인수 기준(AC) 작성 가이드

인수 기준(Acceptance Criteria)은 스토리가 "완료"된 상태를 사용자 관점에서 정의한다.
개발자가 자동화 테스트를 작성하고, PM이 인수 테스트에 사용하는 기준이 된다.

---

## 기본 형식

```gherkin
Scenario: [시나리오 이름 — 행동을 한 줄로 요약]
    Given [사전 조건 — 테스트가 시작될 때의 상태]
    When  [사용자 행동 — 무엇을 했을 때]
    Then  [기대 결과 — 어떤 일이 일어나야 하는가]
```

### 각 구성요소 설명

| 구성요소 | 역할 | 작성 원칙 |
|---------|------|---------|
| **GIVEN** | 테스트 시작 시점의 상태 | 구체적인 사전 조건 (로그인 여부, 데이터 존재 여부 등) |
| **WHEN** | 사용자가 수행하는 행동 | 단일 행동으로 표현 |
| **THEN** | 행동 후 기대되는 결과 | 검증 가능하고 측정 가능하게 작성 |

---

## 작성 원칙

### 1. 테스트 가능해야 한다

AC는 자동화 테스트로 작성할 수 있어야 한다.
"사용자가 만족한다"처럼 측정 불가능한 표현은 피한다.

### 2. 한 AC에 하나의 시나리오

여러 행동과 결과를 하나의 AC에 섞지 않는다.

### 3. 해피 패스 먼저, 예외 케이스 나중

정상 흐름을 먼저 정의한 후 엣지 케이스와 오류 케이스를 추가한다.

### 4. 사용자 관점으로 작성

기술 구현 방식이 아닌, 사용자가 경험하는 것을 기준으로 작성한다.

---

## 예시 비교

### 해피 패스 (정상 흐름)

```gherkin
✅ 좋은 예

Scenario: PDF 다운로드 성공
    Given 제안서 요약 페이지를 방문했을 때
    When  "PDF 다운로드" 버튼을 클릭하면
    Then  영업 제안서의 PDF 파일이 내 컴퓨터에 다운로드된다
```

```gherkin
❌ 나쁜 예

Scenario: 다운로드
    Given 사용자가 로그인했을 때
    When  버튼을 클릭하고 API가 호출되면
    Then  파일이 생성되어 저장된다
```

나쁜 예의 문제:
- GIVEN이 너무 광범위함 (어떤 페이지인지 불명확)
- WHEN에 기술 구현(API 호출)이 포함됨
- THEN이 사용자 관점이 아닌 시스템 관점

## AC 작성 순서

### 1단계: 해피 패스

```gherkin
Scenario: 로그인 성공
    Given 유효한 이메일과 비밀번호로 로그인 페이지를 방문했을 때
    When  로그인 버튼을 클릭하면
    Then  대시보드 페이지로 이동하고 내 이름이 표시된다
```

### 2단계: 엣지 케이스

```gherkin
Scenario: 주문 목록이 비어 있는 경우
    Given 주문 목록이 비어 있는 상태에서
    When  주문 목록 페이지를 방문하면
    Then  "아직 주문 내역이 없습니다" 메시지가 표시된다
```

### 3단계: 오류 케이스

```gherkin
Scenario: 이메일 형식 오류
    Given 이메일 형식이 올바르지 않은 값을 입력했을 때
    When  로그인 버튼을 클릭하면
    Then  "올바른 이메일 형식을 입력해주세요" 오류 메시지가 표시된다
```

## 완성된 AC 세트 예시

### 스토리: 영업 제안서 PDF 다운로드

```gherkin
Scenario: PDF 다운로드 성공
    Given 제안서 요약 페이지를 방문했을 때
    When  "PDF 다운로드" 버튼을 클릭하면
    Then  영업 제안서의 PDF 파일이 내 컴퓨터에 다운로드된다

Scenario: 파일명 형식 확인
    Given 제안서 요약 페이지를 방문했을 때
    When  "PDF 다운로드" 버튼을 클릭하면
    Then  다운로드된 파일의 이름은 "proposal-[고객사명]-[날짜].pdf" 형식이다

Scenario: PDF 생성 중 로딩 상태
    Given PDF 생성에 시간이 걸리는 경우
    When  "PDF 다운로드" 버튼을 클릭하면
    Then  로딩 스피너가 표시되고 버튼이 비활성화된다

Scenario: PDF 생성 실패
    Given PDF 생성 중 서버 오류가 발생했을 때
    When  다운로드를 시도하면
    Then  "PDF 생성에 실패했습니다. 다시 시도해주세요" 메시지가 표시된다
```

## 자주 하는 실수

| 실수 | 예시 | 해결 방법 |
|------|------|---------|
| 검증 불가한 결과 | `THEN 사용자가 만족한다` | 구체적인 UI 변화로 표현 |
| 여러 행동 묶기 | `WHEN 버튼을 클릭하고 입력하면` | AC를 분리 |
| 기술 구현 포함 | `WHEN API가 200을 반환하면` | 사용자 행동으로 표현 |
| GIVEN 생략 | GIVEN 없이 WHEN부터 시작 | 항상 명확한 사전 조건 명시 |
| 너무 광범위한 THEN | `THEN 모든 것이 정상 작동한다` | 구체적으로 무엇이 어떻게 되는지 명시 |

## UI 스토리 AC 작성 패턴

UI 인터랙션 흐름 단위로 분리된 스토리는 다음 규칙을 따른다.

### 규칙 1: GIVEN에 화면 이름을 명시한다

어느 화면에서 시작하는지 항상 명시한다.

```
✅ GIVEN 메인 화면이 열려 있을 때
✅ GIVEN 에이전트 편집 리스트 화면이 열렸을 때
✅ GIVEN 에이전트 등록 윈도우가 열렸을 때
❌ GIVEN 사용자가 로그인했을 때   ← 화면이 불명확
```

### 규칙 2: THEN에 화면 위치와 보이는 요소를 명시한다

무엇이 어디에 보이는지 구체적으로 작성한다.

```
✅ THEN 오른쪽 상단에 + 버튼이 보인다
✅ THEN 에이전트 이름 입력 필드에 기존 이름이 채워져 있다
✅ THEN 각 에이전트 항목 오른쪽에 수정 버튼과 삭제 버튼이 보인다
❌ THEN 버튼이 표시된다            ← 위치 불명확
❌ THEN 필드가 채워진다            ← 어떤 필드인지 불명확
```

### 규칙 3: 화면 전환은 출발점과 도착점을 모두 명시한다

```
✅ GIVEN 메인 화면이 열려 있을 때
   WHEN + 버튼을 클릭하면
   THEN 에이전트 등록 윈도우가 새로 열린다
```

### UI AC 완성 예시

```gherkin
Scenario: 에이전트 등록 폼 열기
    Given 메인 화면이 열려 있을 때
    When  + 버튼을 클릭하면
    Then  에이전트 등록 윈도우가 새로 열린다

Scenario: 등록 폼 필드 확인
    Given 에이전트 등록 윈도우가 열렸을 때
    When  윈도우를 볼 때
    Then  에이전트 이름 입력 필드와 시스템 프롬프트 입력 필드가 보인다

Scenario: 등록 폼 버튼 확인
    Given 에이전트 등록 윈도우가 열렸을 때
    When  윈도우를 볼 때
    Then  저장 버튼과 취소 버튼이 보인다
```

## 영속성 AC 패턴

데이터를 **생성·수정·삭제**하는 스토리에는 반드시 영속성 AC를 추가한다.
별도의 "영속성 스토리"를 만들지 않고, 각 스토리의 마지막 AC로 포함한다.

### 추가 기준

| 스토리 유형 | 영속성 AC 필요 여부 |
|------------|------------------|
| 데이터 저장 (Create) | ✅ 필수 |
| 데이터 수정 (Update) | ✅ 필수 |
| 데이터 삭제 (Delete) | ✅ 필수 |
| 화면 조회 / 버튼 노출 | ❌ 불필요 |
| 화면 전환 / 취소 | ❌ 불필요 |

### 영속성 AC 형식

```gherkin
Scenario: 앱 재실행 후 저장 데이터 유지
    Given [데이터 변경 작업이 완료된 후] 앱을 종료했을 때
    When  앱을 다시 실행하면
    Then  [변경 내용이 그대로 반영된 상태]가 보인다
```

### 영속성 AC 예시

```gherkin
Scenario: 앱 재실행 후 저장 에이전트 유지
    Given 에이전트가 저장된 후 앱을 종료했을 때
    When  앱을 다시 실행하면
    Then  메인 화면에 해당 에이전트 이름이 리스트에 그대로 보인다

Scenario: 앱 재실행 후 수정 에이전트 유지
    Given 에이전트가 수정 저장된 후 앱을 종료했을 때
    When  앱을 다시 실행하고 에이전트 편집 리스트 화면을 볼 때
    Then  변경된 이름과 시스템 프롬프트가 그대로 보인다

Scenario: 앱 재실행 후 삭제 에이전트 미노출
    Given 에이전트가 삭제된 후 앱을 종료했을 때
    When  앱을 다시 실행하고 에이전트 편집 리스트 화면을 볼 때
    Then  삭제된 에이전트가 목록에 보이지 않는다
```

## TrackerBoot 등록 형식

TrackerBoot의 `tasks` 배열에 각 AC를 한 줄로 매핑한다.

```json
{
  "tasks": [
    {
      "description": "GIVEN 제안서 요약 페이지를 방문했을 때 WHEN 'PDF 다운로드' 버튼을 클릭하면 THEN PDF 파일이 다운로드된다"
    },
    {
      "description": "GIVEN PDF 생성 중 오류 발생 시 WHEN 다운로드를 시도하면 THEN 오류 메시지가 표시된다"
    }
  ]
}
```
