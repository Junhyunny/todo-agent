# 스토리 분리 가이드

사용자 시나리오를 좋은 유저 스토리 단위로 나누기 위한 기준을 정의한다.

---

## 좋은 유저 스토리의 조건

- 팀 전체가 스토리를 읽고 사용자에게 어떤 가치를 제공하는지 이해할 수 있다
- IPM에서의 스토리 논의가 원활하며, 전반적인 방향 자체보다는 세부 사항을 명확히 하는 데 집중된다
- 개발자가 스토리를 완료했을 때를 명확히 알 수 있다
- 스토리가 전달된 후 쉽게 인수할 수 있다
- 팀이 매주 여러 스토리를 완료한다
- 대부분의 백로그 스토리가 독립적으로, 병렬로 완료될 수 있다
- 스토리가 구현 방법이 아닌 무엇을 만들어야 하는지를 설명한다

---

## INVEST 원칙

**INVEST**는 좋은 유저 스토리의 특성을 기억하는 약어다:

| 원칙 | 설명 |
|------|------|
| **I**ndependent (독립적) | 다른 스토리에 의존하지 않고 독립적으로 전달할 수 있어야 한다 |
| **N**egotiable (협상 가능) | 기능에 대한 구체적인 계약이 아닌, 가치 전달에 대한 공유된 이해의 경량 문서다 |
| **V**aluable (가치 있는) | 사용자에게 실질적인 가치를 전달해야 하며, 기술적 수평 분할이 아니어야 한다 |
| **E**stimatable (추정 가능) | 개발자가 복잡도나 노력을 어느 정도 추정할 수 있어야 한다 |
| **S**mall (작은) | 하나의 이터레이션 안에서 완료할 수 있을 만큼 작아야 한다 |
| **T**estable (테스트 가능) | 아직 테스트가 작성되지 않았더라도, 원칙적으로 테스트가 가능해야 한다 |

## 분리 패턴

### 1. 흐름 단계별 분리 (Workflow Steps)

사용자 흐름의 각 단계를 독립 스토리로 분리한다.

**예시 시나리오:** "사용자가 상품을 장바구니에 담고 결제를 완료한다"

```
✅ 장바구니에 상품 담기
✅ 장바구니에서 상품 수량 변경/제거
✅ 결제 수단 선택 및 결제 완료
✅ 주문 완료 확인 이메일 수신
```

### 2. 역할별 분리 (User Roles)

동일한 기능이라도 다른 페르소나가 사용하는 경우 분리한다.

**예시:**
```
✅ 고객이 주문 내역을 조회한다
✅ 관리자가 전체 주문 목록을 조회한다
```

### 3. 해피 패스 우선 분리 (Happy Path First)

핵심 흐름을 먼저 스토리로 만들고, 예외 처리는 별도 스토리로 분리한다.

**예시:**
```
✅ [핵심] 사용자가 이메일로 로그인한다
✅ [예외] 비밀번호 찾기 및 재설정
✅ [예외] 계정 잠금 해제
```

### 5. UI 인터랙션 흐름별 분리 (UI Interaction Flow)

UI가 중심인 기능은 사용자의 조작 흐름을 따라 스토리를 나눈다.
각 스토리는 **사용자가 무언가를 본다**, **클릭하면 반응이 있다**, **입력하고 저장한다** 중 하나에 집중한다.

**분리 기준:**

| 유형 | 설명 | 예시 |
|------|------|------|
| **진입점 노출** | 사용자가 기능을 시작할 수 있는 UI 요소가 보인다 | 오른쪽 상단에 + 버튼이 보인다 |
| **화면 전환** | 버튼/링크를 누르면 다른 화면이나 윈도우가 열린다 | + 버튼을 누르면 등록 윈도우가 열린다 |
| **폼 구성 확인** | 열린 화면에서 필요한 입력 필드와 버튼이 보인다 | 윈도우에 이름 필드, 저장/취소 버튼이 보인다 |
| **데이터 저장** | 입력 후 저장하면 결과가 특정 화면에 반영된다 | 저장하면 목록에 새 항목이 나타난다 |
| **취소/복귀** | 취소하거나 이전 버튼을 누르면 되돌아간다 | 취소 버튼을 누르면 윈도우가 닫힌다 |
| **기존 데이터 노출** | 수정 진입 시 기존 값이 채워져 있다 | 수정 윈도우에 기존 이름이 채워져 있다 |
| **삭제 확인** | 삭제 전 확인 메시지가 표시된다 | 삭제 버튼을 누르면 확인 메시지가 보인다 |

**예시 시나리오:** "에이전트를 등록하고 목록에서 확인한다"

```
✅ 메인 화면 오른쪽 상단에 + 버튼이 보인다                      ← 진입점 노출
✅ + 버튼을 누르면 등록 윈도우가 열린다                          ← 화면 전환
✅ 등록 윈도우에 이름 필드, 시스템 프롬프트 필드, 저장/취소 버튼이 보인다  ← 폼 구성 확인
✅ 값을 입력하고 저장하면 메인 화면에 에이전트 이름이 보인다       ← 데이터 저장
✅ 취소 버튼을 누르면 윈도우가 닫히고 목록에 변화가 없다          ← 취소
```

---

### 4. 데이터 범위별 분리 (Data Variations)

같은 행동이지만 데이터의 경우의 수가 큰 경우 분리한다.

**예시:**
```
✅ 신용카드로 결제한다
✅ 간편결제(카카오페이/토스)로 결제한다
```

## 분리하지 말아야 할 것 (안티패턴)

### ❌ 기술 레이어별 수평 분리

```
// 잘못된 예
- 프론트엔드에서 장바구니 UI를 만든다
- 백엔드에서 장바구니 API를 만든다
- 데이터베이스에 장바구니 테이블을 만든다

// 올바른 예
- 사용자가 상품을 장바구니에 담을 수 있다
```

### ❌ 지나치게 세분화 (단, UI 인터랙션 흐름은 예외)

하나의 스토리가 비즈니스 맥락 없이 "버튼 하나 추가" 수준이라면 너무 작다.
팀이 하나의 이터레이션에서 완료 가능한 의미 있는 기능 단위로 유지한다.

**단, UI 인터랙션 흐름별 분리(아래 패턴 5 참고)는 예외다.**
사용자가 보는 것·클릭하는 것·입력하고 저장하는 것은 각각 독립적인 인수 기준을 가지므로
별도 스토리로 나누는 것이 적절하다.

### ❌ 지나치게 큰 에픽

여러 이터레이션이 필요할 만큼 큰 스토리는 분리한다.
"사용자가 전체 쇼핑 경험을 완료한다"는 에픽이지 스토리가 아니다.

## 분리 결과 검토 체크리스트

각 분리된 스토리에 대해 다음을 확인한다:

- [ ] 이 스토리만 독립적으로 배포해도 사용자에게 가치가 있는가?
- [ ] 인수 기준을 작성할 수 있는가?
- [ ] 하나의 이터레이션(1~2주) 안에 완료 가능한가?
- [ ] "구현 방법"이 아닌 "무엇을 하는지"를 설명하는가?
- [ ] 다른 스토리가 완료되지 않아도 이 스토리를 시작할 수 있는가?
