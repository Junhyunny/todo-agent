# 사용자 스토리 형식 가이드

XP(익스트림 프로그래밍)에서 사용자 스토리는 사용자의 관점에서 기능의 가치를 표현한다.
이 가이드는 "As a / I want / so that" 형식으로 스토리를 작성하는 방법을 설명한다.

---

## 기본 형식

```
As a [페르소나],
I want to [기능/행동],
so that [사용자 가치 — 앱 외부에서 일어나는 일].
```

### 각 구성요소 설명

| 구성요소 | 역할 | 작성 원칙 |
|---------|------|---------|
| **As a** | 이 기능을 사용하는 사람 | 역할이나 상황을 구체적으로 표현 |
| **I want to** | 사용자가 하고 싶은 것 | 기술 구현이 아닌 사용자 행동으로 표현 |
| **so that** | 이 기능을 원하는 이유 | 앱 바깥에서 일어나는 실제 가치 |

---

## As a — 페르소나 작성

### 원칙

페르소나는 이 기능을 사용하는 실제 사람을 나타낸다.
모호한 "사용자" 대신 역할, 상황, 목표가 담긴 구체적인 표현을 사용한다.

### 예시 비교

| ❌ 나쁜 예 | ✅ 좋은 예 |
|----------|---------|
| `사용자` | `영업 담당자 Alex` |
| `고객` | `처음 구매하는 신규 고객` |
| `관리자` | `여러 팀을 관리하는 프로젝트 매니저` |
| `앱 사용자` | `모바일로 주문하는 단골 고객` |

### 페르소나 표현 방식

```
# 이름과 역할
As a 영업 담당자 Alex,

# 상황과 역할
As a 여러 프로젝트를 동시에 담당하는 팀 리더,

# 신규/기존 사용자 구분
As a 처음 서비스에 가입한 신규 사용자,
```

---

## I want to — 원하는 것 작성

### 원칙

사용자가 *하고 싶은 행동*을 표현한다.
기술적 구현 방법이 아니라, 사용자 관점의 동작으로 작성한다.

### 예시 비교

| ❌ 나쁜 예 | ✅ 좋은 예 |
|----------|---------|
| `REST API를 호출해서 데이터를 가져오기를` | `내 주문 목록을 확인하기를` |
| `DB에 레코드를 저장하기를` | `새 고객 정보를 등록하기를` |
| `PDF 파일을 생성하기를` | `영업 제안서의 PDF 버전을 다운로드하기를` |
| `JWT 토큰을 갱신하기를` | `자동으로 로그인 상태를 유지하기를` |

---

## so that — 사용자 가치 작성

### 핵심 원칙

**"so that" 부분은 앱 바깥에서 일어나는 일을 설명해야 한다.**

이것이 가장 중요한 부분이다. 앱 내부의 기능 설명이 아니라,
사용자가 이 기능을 통해 *실제 세계에서* 얻는 가치를 표현한다.

### 예시 비교

| ❌ 나쁜 예 (앱 내부) | ✅ 좋은 예 (앱 외부 가치) |
|-------------------|----------------------|
| `목록 화면을 볼 수 있도록` | `어떤 주문이 처리 중인지 파악할 수 있도록` |
| `알림을 받을 수 있도록` | `중요한 마감을 놓치지 않도록` |
| `PDF를 다운로드할 수 있도록` | `잠재 고객에게 이메일로 제안서를 보낼 수 있도록` |
| `프로필을 수정할 수 있도록` | `팀원들이 나를 올바르게 연락할 수 있도록` |

### 테스트: "그래서 뭐가 좋은데?" 질문

"so that" 부분을 작성한 후 "그래서 사용자에게 뭐가 좋은데?"라고 물어본다.
아직 앱 기능 설명이 나온다면 다시 한 번 더 깊이 파고든다.

```
❌ so that 내 주문 목록을 볼 수 있도록
→ "그래서?" → 어떤 주문이 처리 중인지 알 수 있다
✅ so that 어떤 주문이 현재 처리 중인지 빠르게 파악할 수 있도록
```

---

## 완성된 스토리 예시

### 예시 1 — 영업 제안서 다운로드

```
As a 영업 담당자 Alex,
I want to 영업 제안서의 PDF 버전을 다운로드하기를,
so that 잠재 고객에게 이메일로 보낼 수 있도록.
```

### 예시 2 — 주문 상태 확인

```
As a 모바일로 주문하는 단골 고객,
I want to 내 주문 현황을 실시간으로 확인하기를,
so that 배송이 언제 도착하는지 알고 약속 시간을 잡을 수 있도록.
```

### 예시 3 — 팀 프로젝트 관리

```
As a 여러 팀을 관리하는 프로젝트 매니저,
I want to 각 팀의 진행 상황을 한 화면에서 보기를,
so that 위험한 프로젝트를 조기에 파악하고 리소스를 조정할 수 있도록.
```

## 스토리 제목 작성

제목은 스토리의 핵심을 30자 이내로 압축한 명사구다.

| ❌ 나쁜 예 | ✅ 좋은 예 |
|----------|---------|
| `PDF 기능 추가` | `영업 제안서 PDF 다운로드` |
| `주문 관련 기능` | `실시간 주문 현황 조회` |
| `사용자 설정` | `알림 수신 설정 관리` |

---

## 자주 하는 실수

| 실수 | 설명 | 해결 방법 |
|------|------|---------|
| 기술 스토리 | "API 엔드포인트를 추가한다" — 사용자 가치 없음 | 사용자가 이 기능으로 무엇을 하는지 찾기 |
| 너무 큰 스토리 | 여러 페르소나, 여러 가치가 섞임 | 한 스토리에 하나의 페르소나, 하나의 가치 |
| "시스템" 페르소나 | "시스템 관리자로서" — 비즈니스 가치 불명확 | 실제 사람의 역할로 변경 |
| 구현 묘사 | "데이터베이스를 조회해서" | 사용자 행동 중심으로 재작성 |
