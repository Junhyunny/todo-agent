# 사용자 스토리 형식 가이드

XP(익스트림 프로그래밍)에서 사용자 스토리는 사용자의 관점에서 기능의 가치를 표현한다.

---

## 기본 형식

```markdown
### Why

**As** [페르소나]
**I want** [기능/행동]
**So that** [사용자 가치 — 앱 외부에서 일어나는 일]

### Acceptance Criteria

​```gherkin
Scenario: [시나리오 이름]
    Given [사전 조건]
    When [사용자 행동]
    Then [기대되는 결과]
​```

**Notes:**
[선택 사항 — 추가 맥락, 제약, 설계 힌트]
```

### 각 구성요소 설명

| 구성요소 | 역할 | 작성 원칙 |
|---------|------|---------|
| **As** | 이 기능을 사용하는 구체적인 페르소나 | 역할, 상황, 맥락이 드러나게 표현 |
| **I want** | 사용자가 하고 싶은 것 | 기술 구현이 아닌 사용자 행동으로 표현 |
| **So that** | 이 기능을 원하는 이유 | 앱 바깥에서 일어나는 실제 가치 |
| **Scenario** | 하나의 인수 시나리오 | 시나리오 이름은 한 줄로 행동을 요약 |
| **Notes** | 추가 맥락 | 선택 사항 — 없으면 생략 가능 |

---

## As — 페르소나 작성

### 원칙

`As`는 이 기능을 사용하는 구체적인 사람이나 사용자 유형을 나타낸다.
시나리오에 드러난 역할과 맥락을 바탕으로 작성하고, 정보가 부족하면 스토리 작성 전에 먼저 확인한다.
모호한 일반 표현보다 역할과 상황이 드러나는 페르소나를 사용한다.

### 예시 비교

| ❌ 나쁜 예 | ✅ 좋은 예 |
|----------|---------|
| `사용자` | `영업 담당자 Alex` |
| `고객` | `처음 구매하는 신규 고객` |
| `관리자` | `여러 팀을 관리하는 프로젝트 매니저` |
| `앱 사용자` | `모바일로 주문하는 단골 고객` |

---

## I want — 원하는 것 작성

### 원칙

사용자가 *하고 싶은 행동*을 표현한다.
기술적 구현 방법이 아니라, 사용자 관점의 동작으로 작성한다.

### 예시 비교

| ❌ 나쁜 예 | ✅ 좋은 예 |
|----------|---------|
| `REST API를 호출해서 데이터를 가져오기를` | `내 주문 목록을 확인하기를` |
| `DB에 레코드를 저장하기를` | `새 고객 정보를 등록하기를` |
| `JWT 토큰을 갱신하기를` | `자동으로 로그인 상태를 유지하기를` |

---

## So that — 사용자 가치 작성

### 핵심 원칙

**"So that" 부분은 앱 바깥에서 일어나는 일을 설명해야 한다.**

앱 내부의 기능 설명이 아니라, 사용자가 이 기능을 통해 *실제 세계에서* 얻는 가치를 표현한다.

### 예시 비교

| ❌ 나쁜 예 (앱 내부) | ✅ 좋은 예 (앱 외부 가치) |
|-------------------|----------------------|
| `목록 화면을 볼 수 있도록` | `어떤 주문이 처리 중인지 파악할 수 있도록` |
| `알림을 받을 수 있도록` | `중요한 마감을 놓치지 않도록` |
| `PDF를 다운로드할 수 있도록` | `잠재 고객에게 이메일로 제안서를 보낼 수 있도록` |

---

## 완성된 스토리 예시

### 예시 1 — 영업 제안서 다운로드

```markdown
### Why

**As** 영업 담당자 Alex
**I want** 영업 제안서의 PDF 버전을 다운로드하기를
**So that** 잠재 고객에게 이메일로 보낼 수 있도록

### Acceptance Criteria

​```gherkin
Scenario: PDF 다운로드 성공
    Given 제안서 요약 페이지를 방문했을 때
    When "PDF 다운로드" 버튼을 클릭하면
    Then 영업 제안서의 PDF 파일이 내 컴퓨터에 다운로드된다
​```

**Notes:**
파일명 형식: proposal-[고객사명]-[날짜].pdf
```

### 예시 2 — 주문 상태 확인

```markdown
### Why

**As** 모바일로 주문하는 단골 고객
**I want** 내 주문 현황을 실시간으로 확인하기를
**So that** 배송이 언제 도착하는지 알고 약속 시간을 잡을 수 있도록

### Acceptance Criteria

​```gherkin
Scenario: 주문 목록 조회
    Given 주문 목록 페이지를 방문했을 때
    When 화면을 바라보면
    Then 각 주문의 상태와 예상 배송일이 목록에 보인다
​```
```

## 스토리 제목 작성

제목은 **"[화면/상황]에서 [주체]가 [무엇을 할 수 있다/본다/한다]"** 형식의 서술형 문장으로 작성한다.
사용자가 어떤 맥락에서 무엇을 경험하는지를 한 문장으로 표현한다.

| ❌ 나쁜 예 (명사구, 너무 압축) | ✅ 좋은 예 (서술형, 맥락 포함) |
|----------|---------|
| `PDF 기능 추가` | `제안서 상세 화면에서 PDF 다운로드 버튼이 보인다` |
| `주문 관련 기능` | `주문 목록 화면에서 각 주문의 실시간 상태가 보인다` |
| `사용자 설정` | `설정 화면에서 알림 수신 여부를 변경할 수 있다` |
| `에이전트 추가 진입점 노출` | `메인 화면에서 에이전트 추가 진입점이 보인다` |

## 자주 하는 실수

| 실수 | 설명 | 해결 방법 |
|------|------|---------|
| 기술 스토리 | "API 엔드포인트를 추가한다" — 사용자 가치 없음 | 사용자가 이 기능으로 무엇을 하는지 찾기 |
| 너무 큰 스토리 | 여러 페르소나, 여러 가치가 섞임 | 한 스토리에 하나의 페르소나, 하나의 가치 |
| "시스템" 페르소나 | "시스템으로서" — 비즈니스 가치 불명확 | 실제 사람의 역할로 변경 |
| 구현 묘사 | "데이터베이스를 조회해서" | 사용자 행동 중심으로 재작성 |
