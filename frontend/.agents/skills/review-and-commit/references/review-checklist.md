# 코드 리뷰 체크리스트

`/review-and-commit` 스킬이 2단계 코드 리뷰에서 사용하는 기준입니다.

---

## 🔐 보안 (Security)

### Critical (🔴)

- [ ] 하드코딩된 비밀번호, API 키, 토큰, 자격증명이 없는가?
- [ ] SQL Injection 취약점이 없는가? (원시 쿼리 문자열 연결 확인)
- [ ] XSS 취약점이 없는가? (사용자 입력 미검증 렌더링 확인)
- [ ] 민감 데이터가 로그에 출력되지 않는가?
- [ ] 인증/인가 없이 접근 가능한 엔드포인트가 없는가?

### Warning (🟡)

- [ ] 입력값 검증이 충분한가? (타입, 범위, 길이 등)
- [ ] 에러 메시지에 내부 구현 정보가 노출되지 않는가?
- [ ] 외부 라이브러리의 알려진 취약점이 없는가?
- [ ] CORS 설정이 너무 관대하지 않은가?

---

## 🔁 중복 및 리팩토링 (Duplication & Refactoring)

### Warning (🟡)

- [ ] 3회 이상 반복되는 동일한 로직 블록이 없는가? (DRY 원칙)
- [ ] 추출 가능한 헬퍼 함수나 유틸리티가 없는가?
- [ ] 단일 함수/메서드가 20줄을 초과하며 복잡한가?
- [ ] 중첩 조건문이 3단계 이상 깊지 않은가?

### Suggestion (🔵)

- [ ] 클래스/함수 명칭이 역할을 명확히 표현하는가?
- [ ] 매직 넘버나 매직 스트링이 상수로 추출되었는가?
- [ ] 조건 표현식이 읽기 쉽게 작성되었는가?

---

## 📦 모듈 분리 (Module Separation)

### Warning (🟡)

- [ ] 단일 파일/클래스가 여러 책임을 가지지 않는가? (SRP 위반)
- [ ] 비즈니스 로직이 컨트롤러/뷰에 직접 작성되지 않았는가?
- [ ] 데이터 접근 로직이 서비스 레이어에 혼재하지 않는가?
- [ ] 유틸리티 함수가 특정 도메인에 종속되지 않는가?

### Suggestion (🔵)

- [ ] 파일이 너무 커서 별도 모듈로 분리가 필요한가? (300줄 기준)
- [ ] 의존성 방향이 올바른가? (도메인 → 인프라 방향 확인)

---

## 💡 더 나은 방향 (Better Direction)

### Suggestion (🔵)

- [ ] 현재 테스트가 엣지 케이스를 충분히 커버하는가?
- [ ] null/undefined/empty 케이스 테스트가 있는가?
- [ ] 에러 케이스 테스트가 있는가?
- [ ] 타입 안전성이 충분한가? (TypeScript: `any` 사용 여부 등)
- [ ] 비동기 코드의 에러 핸들링이 적절한가?
- [ ] 더 간결하거나 관용적인 표현이 있는가?
- [ ] 성능에 명백히 문제가 되는 패턴이 없는가? (N+1 쿼리 등)

---

## 심각도 판단 기준

| 레벨 | 기준 | 처리 |
|------|------|------|
| 🔴 Critical | 보안 취약점, 데이터 손실 가능성, 서비스 중단 위험 | 사용자에게 계획 제시, 반드시 수정 권고 |
| 🟡 Warning | 코드 품질 저하, 유지보수 어려움, 잠재적 버그 | 사용자에게 계획 제시, 수정 권고 |
| 🔵 Suggestion | 더 나은 패턴, 가독성 개선, 테스트 강화 | 사용자에게 계획 제시, 선택적 수정 |
